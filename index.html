1-tema

Sapanı támiynlew usılları: Ónim hám process sapası

SQA (Software Quality Assurance) — bul programmalıq ónimi rawajlanıw basqıshlarındaǵı sapalı processlerdi támiynlew. Process sapası - jumıs usıllarınıń durıslıǵı, ónim sapası - final onımi talaplarǵa say keletuǵının ańlatadı.

1-súwret: Process penen ónim sapası bir-birin tolıqtıradı.
"Sapalı process — sapalı ónimdiń kópillik kefili" — Deming principi

Qosımsha maǵlıwmat: Process sapasın jaqsılaw ushın PDCA (Plan-Do-Check-Act) cikli qollanıladı. Ónim sapası bolsa funksional talap hám performance kórsetkishler arqalı ólshenedi.

2-tema

Sapanı qadaǵalaw (QC) basqıshları

QC tómendegi 4 tiykarǵı bosqısh tan ibarat: 1) Talap analizi; 2) Kod review hám statikalıq analiz; 3) Dinamikalıq testlew; 4) Defektlerdi saplaw hám esabat.

QC procesi ónimniń hár bir basqıshında qátelerdi anıqlaw hám olardı saplawǵa baǵdarlanǵan. Bul bosqıshlarda statikalıq analiz ushın SonarQube, dinamikalıq test ushın JUnit hám Selenium qollanıladı.

3-tema

Nátiyjelilik hám isenimlilik kórsetkishleri

CPU nátiyjeliligi: 42% jaqsılandı
MTBF: 850 saat
MTTF: 720 saat
Qálpe kórsetkish: 0.9%

MTBF (Mean Time Between Failures) — eki qáte ortasındaǵı ortasha waqıt. Joqarı MTBF isenimlilikke ısarat. Nátiyjelilik bolsa resurslardı unamlı paydalanıw (CPU, xotira). MTTF (Mean Time To Failure) — birinshi qátege shekemǵi waqıt. Bul kórsetkishler sistemalardıń isenimliligin bahalawda tiykarǵı rol oynaydı.

Máselen, bankomat sistemasında MTBF 1000 saat dan joqarı bolıwı kerek.

5-tema

Sanlı bahalaw: defekt tıǵızlıǵı, kod qamrovı

Kod qamrovı ≥80% dep maqul esaplanadı. Siklik murakkablıq Joó (McCabe) úlkenligi quramalılıqtı kórsetedi. Defekt tıǵızlıǵı 1 KLOC (mıń qatar kod) ushın 1.8 dep kórsetilgen.

Eger defekt tıǵızlıǵı 3 ten joqarı bolsa, kod qayta kórip shıǵılıwı kerek.
6-tema

Sapa kriteriyaları: tolıqlıq, anıqlıq, turaqlılıq

  • Tolıqlıq (Completeness) — ónim barlıq funksiyalardı orınlaydı, hesh qanday kemislik joq.
  • Anıqlıq (Accuracy) — mısalı, finanslıq sistemalarda qáte 0.01% ten tómen bolıwı shárt.
  • Turaqlılıq (Robustness) — júk kórgen sáyiste (stress-test) buzılmaydı, qáte situaciyalarda da isleydi.
  • Qayta isletiwshilik — kod modulleri basqa joybarlarda da qollanıla aladı.
7-tema

Standartlastırıw: ISO 12207, CMMI, IEEE 829

ISO/IEC 12207 — programmalıq támiynat ómir cikli processlerin belgileydi. CMMI (Capability Maturity Model Integration) — process jetiliskenlik dárejelerin (1-den 5-ge shekem) bahalaydı. IEEE 829 — test dokumentaciyası (test reje, test case, qáte esabatı) ushın standart.

8-tema

Sertifikatlash : DO-178C, meditsina hám bank sapası

Aviaciya (DO-178C), medicina qurılmaları (IEC 62304), bank sistemaları (PCI DSS, ISO 27001) ushın sertifikatsiyasız bozarǵa shıǵıw qıyın. ISO 9001 (Jámi basqarıw) hám ISO 27001 (maǵlıwmatlar qáwipsizligi) de úzliksiz talap etiledi.

DO-178C dárejesi A (critical) dárejesinde hár bir kod qatarı test qamrovın talap etedi.
9-tema

TQM: Úzliksiz jetilistiriw (Kaizen)

Jámi sapalı basqarıw — demokratik liderlik, processlerdi optimallastırıw. Hár bir sprint ten keyin retrospektiv ótkeriw, client qanaatlanıwın ólshew.

"Búgin keshigiden jaqsıraq bolıw — TQM principi"

TQM tiykarında PDCA cikli, statistikalıq process basqarıw (SPC), benchmark hám toparlı jumıs metodları bar.

10-tema

Verifikatsiya vs Validatsiya

  • Verifikatsiya: “Tuwrı qurıp atamız ba?” (specifikaciyaǵa saylıq) — statikalıq analiz, kod review, inspection.
  • Validatsiya: “Tuwrı ónim qurdıq pa?” (paydalanıwshı mútájligi) — dinamikalıq test, betatesting, alfa/beta versiyalar.

Máselen, specifikaciya boyınsha túyme basqanda jaqtılıq jánedi — verifikatsiya. Paydalanıwshı usı funkciyanı qalay qabıl etedi — validatsiya.

11-tema

CI/CD quwırları (DevOps)

Continuous Integration / Continuous Deployment : Jenkins, GitHub Actions, GitLab CI, Azure DevOps. Avtomatlastırılǵan build, test hám deploy sapalıqtı tezletedi.

CI/CD pipeline'da unit test, integraciyalıq test, sonardıq endi statikalıq analiz avtomat túrde iske tusedi.

12-tema

Agile / Scrum arqalı basqarıw

Sprintlar (1-4 hápte), product backlog, daily stand-up. Sapalıq ushın test avtomatizaciyası hár bir sprintke kiredi. Scrumda Definition of Done (DoD) — sprint ta'minlenetuǵın sapalıq standartlar jıynaǵı.

13-tema

Shegaralıq analiz (BVA) hám ekvivalentlik bóliniw

Mısal: parol uzınlıǵı 8–16 simvol. Sınaw mánisleri: 7,8,9,15,16,17 (shegaralıq mánisler). Jáne Qara qutı (Black Box) – input/outputqa qaray, kod ishki strukturası kórinbeydi. Aq qutı (White Box) – kod logikasın analiz qıladı, branch hám path test.

Ekvivalentlik bólistiriw: 1-7 (qábilet joq), 8-16 (qábilet bar), 17+ (qábilet joq). Testlewde hám bir diapazondan bir mánis alınsa boladı.

14-tema

Test rejesi quramı: test strategiyası, resurslar, test kriteriyaları, test jumısları, shıǵıw kriteriyaları. Selenium, Cypress, Postman, JUnit járdeminde unit test, integraciyalıq test, E2E test.

Unit test (Jest/JUnit)
Integraciya (Postman)

15-tema

Huqıqıy tiykarlar hám AI testlewge integraciya

Programmalıq támiynat sapası boyınsha huqıqıy hújjetler: ISO normativleri (ISO 9001, ISO 27001), milliylestiriw (GOST), cifrlıq huqıq (GDPR, CC). AI járdeminde test generation (diffblue, testim.io), CMMI dárejelerin kóteriw.

"Keleshekte sapalıq testlew AI hám ML modelleri menen birge isleydi: avtomat test case generation, qáte boljaw, test maǵlıwmatların sintetizaciyalaw."

Machine Learning modellerin testlew (ML Test Score) hám AI etikası da sapa standartlarınıń bir bólegine aylanbaqta.
